# MemoriaKids 技術構想レビュー — AI/IT戦略コンサルティング評価

**レビュアー**: AI/IT戦略コンサルタント（元トップティアコンサルファーム、AI/MLシステム設計・DX専門）

---

## 1. AI/MLアーキテクチャの技術的妥当性

**妥当な判断:** RetinaFace + ArcFace（InsightFace）は2026年でもデファクト。ONNX Runtime + Faiss HNSW も合理的。

**技術的懸念:**

| 項目 | 深刻度 |
|------|--------|
| mAP@0.5 ≥ 99.2%は幼児顔で達成困難（成人比10〜15%低下） | **Critical** |
| Few-shot 5枚は楽観的（15〜20枚が現実的） | **High** |
| Cross-Age TAR@FAR=1e-4 ≥ 95%は0〜6歳児で非現実的 | **High** |
| Min Face Size 40x40pxは集合写真で不足 | **Medium** |

**根本的な問い:** 保育園にGPU搭載デスクトップPCを設置する運用モデル自体が、保育現場のリアリティに合致しているか？

---

## 2. Federated Learningの実現可能性

**評価: Low-Medium。Phase 1-2のロードマップから完全に除外すべき。**

- ResNet-100（65Mパラメータ）の勾配転送は保育園回線（上り10〜50Mbps）で非実用的
- Non-IID問題が深刻（園ごとの年齢構成・撮影環境の差異）
- ε=8は幼児顔データに対して緩すぎる（ε≤4が社会的水準）
- 暗号化embeddingのみクラウド送信で既にプライバシー担保可能 → FL は過剰投資

---

## 3. 分散ストレージの技術的課題

- **復元レイテンシ問題**: 4フラグメントの収集・復号・再構成に数秒〜十数秒 → UX致命的
- **IPFS準拠のノード可用性**: 保育園PC稼働は7:00〜19:00。夜間・休日にノード50%以上オフライン
- **Phase 1-2はクラウドストレージをプライマリとし、分散はPhase 3以降にコスト最適化手段として段階導入すべき**

---

## 4. セキュリティアーキテクチャ

**強み:** Defense in Depth 5層モデル、FIDO2/Passkey、Crypto Shredding、mTLS + WireGuard

**課題:**
- **「顔ベクトルは復元不可能」は技術的に不正確** — NbNet等で近似再構成が実証済み。Cancelable Biometrics の検討が必要
- 子どもの生体データの法的位置づけ（個人識別符号該当性）の弁護士確認が必須

---

## 5. 技術的負債のリスク

- TypeScript/Python二重スタック → 型定義の二重管理
- **データストア5種は5〜10名チームで維持不可能** → PostgreSQL + pgvector + Redis + S3 に統合推奨
- Edge + Cloud二重環境 → 通常SaaS開発の2倍以上の工数

---

## 6. 改善提案（優先度順）

| 優先度 | 提案 |
|--------|------|
| **P0** | MVPの技術スコープ大幅縮小（クラウドファースト） |
| **P0** | 顔認識KPIの現実的再設定（Phase 1: Precision≥90%, Recall≥85%） |
| **P1** | MLパイプラインのCI/CD設計追加（Canaryデプロイ、ロールバック） |
| **P1** | データストアの統合・削減（PostgreSQL + pgvector で Faiss排除） |
| **P2** | ライセンス戦略見直し（コア暗号化: AGPL、フロント: MIT、モデルウェイト: 非公開） |
| **P3** | 子ども生体データの法的・倫理的フレームワーク策定 |

**総括:** コアバリュー「写真サブスク × AI自動キュレーション」の仮説検証を最小技術スコープで最速実行すべき。Edge AI・分散ストレージ・FLはPMF確認後に段階導入。
