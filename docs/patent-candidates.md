# MemoriaKids 特許取得準備候補

> 作成日: 2026-04-09
> ステータス: 検討中

---

## 概要

MemoriaKids の技術的新規性を保護するための特許出願候補をまとめる。
保育×AI×分散インフラの組み合わせにより、競合他社にない独自技術を多数保有している。

**出願優先順位: 1 > 3 > 4 > 5**

---

## ⭐⭐⭐⭐⭐ 最有力候補

### 候補1: 音声AI連絡帳 + マイルストーン自動除外フィルタ

**優先度: 最高（第1位）**

#### 技術概要

保育士の会話音声から園児別の連絡帳ドラフトを自動生成するパイプライン。

```
音声録音 → Whisper文字起こし → 園児別に自動分離 → 連絡帳ドラフト自動生成
```

#### 処理フロー

1. **音声アップロード** — MP3/WAV/M4A対応（100MB上限）、status: `uploading`
2. **文字起こし** — Whisper-large-v3 による全文トランスクリプト生成、status: `transcribing`
3. **園児別抽出** — 発話内容を園児ごとに自動分離（childExtracts）、status: `extracting`
4. **ドラフト生成** — ケアノート参照 + マイルストーンフィルタ適用、status: `generating`
5. **先生レビュー** — ドラフト確認・編集・承認、status: `review_pending` → `completed`

#### 核心の新規性: マイルストーン自動除外フィルタ

「初めて歩いた」「初めて○○できた」等の**発達マイルストーンを連絡帳から自動除外**する仕組み。

- **児童心理学的配慮**: 家庭での"初体験"を尊重し、保護者が最初に喜びを感じる機会を奪わない
- **DEFAULT_MILESTONE_KEYWORDS**: デフォルトの除外キーワード定義
- **カスタムフィルタ**: care_notes の `milestone_filter` カテゴリで園ごとにカスタマイズ可能
- **filteredItems**: 除外された項目と除外理由を記録し、先生が確認・復元可能
- **source管理**: `ai_generated`（完全自動）→ 編集後 `ai_assisted`（AI補助）に変更

#### 特許性の根拠

- 「保育×AI」領域で類似技術なし
- 技術的解決手段（NLP + フィルタリング）と社会的課題（保護者心理への配慮）の組み合わせ
- 審査通過の可能性が最も高い

#### 関連ファイル

- `backend/src/routes/audio.ts`
- `backend/src/db/schema.ts`（audio_recordings, audioTranscripts, contact_book_entries）

### 候補2: 分散暗号化バックアップ + GPU計算還元モデル

**優先度: 最高（特許取得済みの可能性あり — 要確認）**

#### 技術概要

保育園のデスクトップPCを暗号化ノードとして活用し、分散バックアップとAI計算を同時に実現するエコシステム。

```
保育園PC（ノード） ←→ AES-256暗号化データ ←→ 3拠点以上に分散保存
                    ↓
            GPU計算リソース提供（Whisper / 顔認識）
                    ↓
            マイニング方式で報酬還元
```

#### 技術構成

- **AES-256暗号化**: 全データを暗号化した上で3拠点以上に分散保存
- **ノード参加型**: 各保育園のPCがストレージ＋計算ノードとして参加
- **GPU活用**: Whisper音声処理・InsightFace顔認識をノード上で分散実行
- **報酬還元**: マイニング方式で計算リソース提供に対する報酬を園に還元
- **データ健全性ダッシュボード**: 分散データの整合性を可視化

#### 核心の新規性

**SaaS利用者がインフラ提供者にもなるエコシステム**

- 従来: SaaS利用者はサービスを消費するだけ
- MemoriaKids: 利用者（保育園）がインフラを提供し、その対価を受け取る
- 月額500円のサブスク収益 → 一部を園に還元 → PCの購入・維持費に充当

#### ビジネスモデルとの統合

```
保護者（月額500円サブスク）
    │
    ├→ 収益の一部を園に還元
    │   ├─ PC購入・維持費（ストレージノード）
    │   └─ GPU計算報酬（マイニング方式）
    │
    └→ 写真無制限 + AI顔認識分類 + 連絡帳AI生成
```

#### 特許性の根拠

- 分散ストレージ + AI計算 + 報酬還元の三位一体モデル
- 保育業界に特化したデータ主権の確保（各園が所有権を維持）
- クラウドストレージ依存からの脱却

## ⭐⭐⭐⭐ 有力候補

### 候補3: マルチモーダル登降園認証統合システム

**優先度: 高（第2位）**

#### 技術概要

5種類の認証方式を統合し、園ごとにカスタマイズ可能な登降園管理システム。

| 認証方式 | 技術 | セキュリティ |
|---------|------|------------|
| **手動入力** | スタッフ操作 | スタッフ認証 |
| **QRコード** | HMAC-SHA256署名 | 署名検証 + 有効期限管理 |
| **BLEビーコン** | UUID + major/minor | RSSI閾値（デフォルト -70dBm）+ ドウェル時間（30秒） |
| **IoTデバイス** | deviceIdentifier | ゲートウェイ認証（X-Gateway-Key + タイミング攻撃対策） |
| **顔認識** | InsightFace + ONNX | 信頼度閾値（デフォルト 0.85） |

#### 核心の新規性

- **園ごとのカスタマイズ**: `nursery_attendance_settings` で有効方式・閾値・時間帯を個別設定
- **兄弟一括登園**: 1操作で複数園児を同時打刻（最大20名）、`SiblingBatchDialog` で追加確認
- **タッチパネルUI**: 汎用タブレット（iPad等）で即導入可能な2段階UI（クラス選択 → 園児選択）
- **方式メタデータ記録**: `methodMeta` (jsonb) に認証方式固有の情報を記録

#### 関連ファイル

- `backend/src/routes/attendance.ts`
- `backend/src/services/attendance.service.ts`
- `backend/src/db/schema.ts`（attendance_records, qr_tokens, ble_beacons, iot_devices）
- `frontend/app/attendance/touchpanel/page.tsx`

### 候補4: AI顔認識による写真自動分類 + サブスク連動アクセス制御

**優先度: 高（第3位）**

#### 技術概要

写真アップロード時にAI顔認識で園児を自動タグ付けし、サブスクリプションプランに応じてアクセスを動的制御するシステム。

#### 処理フロー

```
写真アップロード → InsightFace顔検出 → 顔ベクトル抽出 → コサイン類似度比較 → 園児自動タグ付け
```

1. **写真保存** — ディスク保存 + DBレコード作成
2. **顔検出** — ONNX Runtime + InsightFace でバウンディングボックス + 埋め込みベクトル取得
3. **顔マッチング** — 既存ベクトルとのコサイン類似度比較（閾値 > 0.85）
4. **タグ付け** — `face_vectors` テーブルに園児ID・ベクトル・信頼度・領域を記録

#### サブスク連動アクセス制御

| プラン | 閲覧解像度 | DL品質 | 月間DL上限 |
|--------|-----------|--------|-----------|
| free（¥0） | 480px | 800px | 10枚 |
| basic（¥4,980） | 1,920px | 1,920px | 100枚 |
| premium（¥9,980） | 3,840px | 3,840px | 無制限 |
| enterprise（¥29,800） | 原画 | 8K原画 | 無制限 |

- **Sharp.js**: 動的画像リサイズ
- **photo_downloads**: DL履歴記録 + 月間制限チェック

#### 核心の新規性

- 顔認識による自動分類とサブスク課金を連動させたアクセス制御モデル
- プランに応じた解像度・枚数の段階的開放

#### 関連ファイル

- `backend/src/services/photo.service.ts`
- `backend/src/routes/photos.ts`
- `backend/src/db/schema.ts`（photos, face_vectors, photo_downloads）

## ⭐⭐⭐ 補強特許

### 候補5: ケアノート連動AI文書生成

**優先度: 中**

#### 技術概要

園児ごとの配慮事項（ケアノート）をAI文書生成に反映し、個別最適化された連絡帳を自動生成するシステム。

#### ケアノートのカテゴリ

| カテゴリ | 例 |
|---------|-----|
| health | 持病・通院情報 |
| allergy | アレルギー情報 |
| behavior | 行動特性 |
| development | 発達段階 |
| family | 家庭環境 |
| dietary | 食事制限 |
| medication | 服薬情報 |
| milestone_filter | マイルストーン除外設定（候補1と連携） |
| communication_style | コミュニケーションスタイル |

#### 核心の新規性

- **文体自動切替**: `communication_style` に基づき、園児ごとに敬語/カジュアルを自動選択
- **優先度制御**: priority（0=低, 1=中, 2=高）により、重要な配慮事項を優先的に反映
- **アクティブ管理**: `isActive` フラグで時期的に不要になった配慮事項を無効化

#### 関連ファイル

- `backend/src/db/schema.ts`（care_notes）
- `backend/src/routes/audio.ts`（ドラフト生成時にケアノート参照）

## 出願戦略

### 推奨スケジュール

| 段階 | 候補 | アクション |
|------|------|-----------|
| **第1弾** | 候補1（音声AI連絡帳） | 最優先で出願準備。「保育×AI」で類似技術なし |
| **第2弾** | 候補3（マルチモーダル認証） | 候補1と同時期に出願検討 |
| **第3弾** | 候補4（写真自動分類+サブスク制御） | 実装の成熟度を見て出願 |
| **補強** | 候補5（ケアノート連動） | 候補1の従属クレームとしても検討可 |

### 候補2について

分散暗号化バックアップ + GPU計算還元モデルは**特許取得済みの可能性あり**。
ステータスを確認の上、未取得であれば第1弾と同時に出願を推奨。

### 注意事項

- 出願前に**先行技術調査**（特許庁J-PlatPat、Google Patents）を実施すること
- 実装コード・DB設計が特許明細書の裏付けとなるため、コードの変更履歴を保全すること
- 候補5は候補1の**従属クレーム**として組み込むことで、出願コストを削減可能
